What is TDD?

üèõÔ∏è A design technique.

üìà A guided evolution.

üß™ Tests first.

  • It a technique in which tests help us design our code, it‚Äôs not really about testing.
  • It is a guided evolution with clear feedback after each step.
  • And last, but not least, it‚Äôs a process where we write tests before we write the production code.

There are 3 steps in TDD.

Red - this is where we start by writing a failing test. It will fail because the code we want to test doesn‚Äôt exist yet. It will fail as long as we don‚Äôt solve the problem we stated in the test.

This approach is the other way around to the test-last approach where we write tests to make them pass. It changes the dynamic between prodcution code and tests. In TDD we don‚Äôt try to prove that the code we wrote works as expected using test. We write code to accomplish what is stated in tests.

‚ùå Red

‚úÖ Green

Write the production code.

Only as much as needed to make tests pass. ‚úã

üîÅ Refactor

  1. Green - once we have at least one failing test, we can write production code to satisfy it.

Since we have tests, we can run them as often as we need. We have a reproducible way to check if the code we write works as expected. We don‚Äôt need to setup the environment by hand each time we sit down to write the code or to do a lot of clicking in the app, if we‚Äôre developing an app.

If the code does what we want it to, tests will pass.

  1. We write only as much production code as needed to make tests pass. Why would we write more code, if tests are already satisfied? Don‚Äôt do it, You Ain‚Äôt Gonna Need It.

  1. Last step is refactoring - Were you ever afraid of refactoring your code because you thought you might break something? TDD helps you mitigate this problem. We have tests that ensure that the code works as expected. So we can be more liberal in changes we make to the code. If written well, tests don‚Äôt constrain us, they empower us to make bold changes.

  2. We don‚Äôt only want to have working code, we want to have good code. We can spend some time in this step to make sure our code is clean and easy to understand.

  1. It‚Äôs useful to remember that each test consists of three parts:
  • Setup
  • Execution
  • Assertion

We can use #Arrange, #Act, #Assert comments to split our tests into those distinct parts.

  1. We start with the setup.

#Arrange is where we prepare the environment. This is where we need to figure out the inputs required to solve the problem.

This is very powerful, why? At this stage, we focus on what we need to be able to solve the problem. Not how we will solve it.

Without understanding the problem we won‚Äôt be able write down the test. And this is good, we won‚Äôt write any production code until we have the understanding of the problem. This helps us mitigate the risk of implementing the wrong solution.

  1. #Act is where we call our code.

TDD gives us the freedom to experiment with interfaces.

We can try out a few different variants even before we start writing the production code, possibly saving a lot of time on refactoring.

The test you write is the first user of your code. The way you call your code from tests is the way you will call it from the production code. So, think about how you want to use your code and then write it down in a test.

  1. In #Assert we focus on how to observe that the code works as expected.

Again, we won‚Äôt be able to do it without understanding the problem. We should know what is the expected output for given inputs.

Although we could skip writing down the assertion, but only for a few iterations of prototyping, to help us undestand the data better, and design the output we want.

So, in TDD we first define inputs and outputs, treating the not-yet-existing code as a black box.

This approach produces tests that don‚Äôt test implementation details. It‚Äôs impossible to do so since there is no implementation yet!

This makes tests focused on the behavior.

How do we use it for more complex problems?

2 approaches

üìà Bottom-up

üìâ Top-down

2 approaches

üìà Bottom-up

Start small. üß±

Use existing pieces of codebase. üß©

A design emerges. üèõÔ∏è

üìâ Top-down

  1. We start small, with the smallest piece of the problem we can solve.
  2. We use existing pieces of codebase.
  3. We don‚Äôt know upfront how the solution will look like. We let the design emerge from the tests and code that we write.

2 approaches

üìà Bottom-up

üìâ Top-down

List high-level objectives first. üéØ

Describe behavior from user perspective. üôå

Fill in the gaps until tests pass. üß©

  1. We focus on high-level objectives first.
  2. We describe the behavior from the user perspective. If it‚Äôs a Shiny app, this is the perspective of the user interacting with the app. If it‚Äôs a package, this is a perspective of the user using the functions exported from the package.
  3. We fill in the gaps in code, until tests pass.

So how do we choose the approach?

üèéÔ∏è We do what helps us move fast.

If we‚Äôre developing a Shiny module a list of user behaviors we want to implement can be more useful. We do top-down approach.

If we‚Äôre developing a computation of a complex algorithm, you might want to start from the smallest piece of the algorithm and build up from there. This would be bottom-up approach.

We do what helps us move fast.

üß™ Tests are the driving force.

They tell us if the code does what we think it does.

  1. This is what Test-Driven means. Tests drive the development process.
  2. They tell us what to do next and if the code we implemented does what we think it does.

So you probably have some requirements from your client that you need to implement. How do we get from there to tests?

  • üëç Examples help us understand the problem.
  • üëé Abstract descriptions don‚Äôt.

Use specific examples. Examples are translatable to tests. Examples help us understand the problem better.

Abstract descriptions don‚Äôt have these properties.

üìù Acceptance criteria

A list of verifiable statements.

  • ‚òëÔ∏è Something should do something.
  • ‚òëÔ∏è Something should do the other thing.
describe("something", {
  it("should do something", {
    # Arrange

    # Act

    # Assert
  })

  it("should do the other thing", {

  })
})

So how do we get these examples? 1. A common way of writing down requirements are acceptance criteria. They are a list of verifiable statements. 2. For example: something should do something. 3. Or it should also do the other thing when something else happens. They should be:

  • Clear, so that everyone understands them.
  • Concise, so that there‚Äôs no ambiguity.
  1. See how easy it is to put those descriptions into a test file, then we can start filling those test.

üìù Test cases

üéØ Specific for low-level (unit) tests.

üí¨ Abstract for high-level (acceptance) tests.

Some tips of writing test cases.

üìù Test cases

üéØ Specific for low-level (unit) tests.

‚òëÔ∏è It should work correctly.

‚òëÔ∏è It should throw an error when the table has 1 row.

üí¨ Abstract for high-level (acceptance) tests.

  1. They should be specific for low-level tests, when we have a specific input and output.
  2. We should be specific for tests of business logic and edge cases.

üìù Test cases

üéØ Specific for low-level (unit) tests.

üí¨ Abstract for high-level (acceptance) tests.

‚òëÔ∏è It should render the table after clicking the ‚ÄúGo‚Äù button.

‚òëÔ∏è It should show the summary.

  1. For high-level tests, as tests of Shiny modules it‚Äôs better to make them abstract, so that they will be true even if the implementation details change.

  2. For tests of a Shiny module, don‚Äôt tie requirements to the UI. Focus on the business goal. Write those requirements such as if you‚Äôve never heard about buttons, dropdowns or inputs.

This is the key to having test cases decoupled from implementation.

The example requirement would no longer be valid if we changed the label of the button, or if we removed the button altogether.

But the business goal would still be valid. It should show the summary, no matter how it can be achieved.

Tests show we solved the problem.

üìù And document what we solved.

So, this means that tests can show that we solved a problem. The will document it, and this documentation will be valid as long as the tests pass, compared to code comments or other documentation types which can get outdated anytime.

Drive communication with a client.

üìä Via executable examples on real data.

And this also means that we can use tests to communicate with our clients better. Tests can empower them to cooperate with us more closely. They can provide us examples which we can translate into tests. Tests are artifacts they can inspect and validate if that‚Äôs what they needed.

TDD x Shiny

  • üì¶ The testable unit of an app is a Shiny module.

  • ‚ú® It represents a feature.

  • üñ•Ô∏è It can be run as a standalone app.

So let‚Äôs see how we can use TDD with Shiny.

There are 3 things we need to know before we start.

  1. The testable uning of a Shiny app is a Shiny module.
  2. A module can represent a feature, and by feature I mean a part or the whole problem we‚Äôre solving with the app.
  3. And that we can easily run modules as standalone Shiny apps.

  1. Next we can focus on the assertion.

I‚Äôm using functions from selenider package that provides a few nice assertions on an object derived from the AppDriver.

  1. Think for a moment what outcome means that you met your objective.

In this case, asserting that the table is visible and that it has at least one row is a good way to ensure that the summary is shown.

Notice how this assertion is in a method. This way we can reuse it in other test cases. If at any point in time, our undestanding changes, we can change the implementation of this method, and all tests will be updated.

Pay attention to the name of the method. It‚Äôs abstract, it doesn‚Äôt mention the UI, that it‚Äôs a table. If we decide to show the summary as a plot, this would still be valid. It‚Äôs the same business goal, we don‚Äôt mention how it‚Äôs achieved.
